# 调试 company-search Skill 学到的 9 条核心教训

> 2026-05-04 | 调试对象：飞书多维表格客户数据查询

---

## 1. 字段名以 API 实际返回为准，不要凭记忆

旧 skill 里写的是 `企业简介`、`企业信息收集` 是两个字段。实际调用 `field_list` 后发现：字段名是 `公司简介`（不是"企业简介"），而 `Contacts` 和 `企业信息收集` 是**两个完全独立的字段**，内容不同。

**每次新建/大幅修改涉及飞书表格的 skill 前，必须先调用一次 `appTableField_list` 拿到真实字段名。**

---

## 2. `automatic_fields: true` 是最优解，手动列字段名注定会漏

旧 skill 显式列了 20 个 field_names，但实际表格有 25 个字段。手动列举要么遗漏、要么拼错。

**用 `automatic_fields: true` 不需要知道字段名，也能返回所有字段的全部内容，是最可靠的方式。**

---

## 3. 空字段在 API 响应里是"不返回该 key"，不是返回空值

当 `企业信息收集` 在飞书表格里是空的，API 响应里**根本没有这个 key**，不是返回 `null` 或空字符串。

```json
// 企业信息收集 为空时，API 返回：
{}

// 而不是：
{"企业信息收集": ""}  // 永远不会这样返回
```

代码里判断空字段要检查 key 是否存在，不能写成 `fields.企业信息收集 === ""`。

---

## 4. 不同字段类型返回的数据结构不同

| 字段类型 | 返回格式 |
|---------|---------|
| Text/多行文本 | `[{"type":"text","text":"..."}]` 或含 `url` 对象的数组 |
| SingleSelect | `"值"`（直接字符串） |
| MultiSelect | `["值1","值2"]`（数组） |
| Number | `123`（数字，不是数组） |
| DateTime | `1774972800000`（毫秒时间戳，需 ÷1000 转换） |
| Url | `[{"type":"url","link":"...","text":"..."}]` |
| Files/Mention | `[{"type":"mention","token":"...","text":"..."}]`，token 可拼成链接 |

---

## 5. Files/Mention 类型字段返回的是飞书文档 token

`Files` 字段里返回的是飞书文档 token，需手动转换：

```json
"Files": [{"type":"mention", "token":"BDatdxUV8oz4Fux2ylBcOn8Rn1g", ...}]
```

**转换为可访问链接**：`https://my.feishu.cn/docx/{token}`

---

## 6. DateTime 字段是毫秒时间戳，需要转换

`创建日期` 返回 `1774972800000`，要除以 1000 再转 `yyyy-MM-dd` 格式。

---

## 7. Contacts ≠ 企业信息收集（最易混淆的点）

| 字段 | 内容 |
|------|------|
| Contacts | 联系人姓名、电话、邮箱、职务、股东信息 |
| 企业信息收集 | 结构化的尽职调查信息（9个问答格式） |

两者名称相似但内容完全不同，容易混淆。

---

## 8. Skill 的"显示逻辑"必须和"数据获取逻辑"一起设计

旧 skill 没有定义"哪些字段不显示"、"空字段怎么标"、"URL 怎么格式化"。这些逻辑应该在 skill 文档里固化下来，否则每次输出格式不一致。

---

## 9. 测试时要用真实数据验证，不能只看 skill 文档

整个调试过程：写 skill → 实际调用 API → 发现返回和预期不符 → 修正 skill。**skill 文档写完不等于对了，必须用真实调用验证。**
